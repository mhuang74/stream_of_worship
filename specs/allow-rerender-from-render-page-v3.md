# Allow Re-render from Render Page (v3)

## Summary

When a songset has already been rendered, navigating to the render page
(`/songsets/[id]/render`) currently shows the `RenderComplete` screen with
download links and no way to change parameters and re-render. This plan changes
the behavior so the render page **always shows the parameter form** when a
previous render exists, with a single-line info banner indicating a previous
render completed. A confirmation dialog (with parameter summary) is shown
before submitting a re-render.

Additionally, a `lastCompletedRenderJobId` field is added to the songsets table
to ensure download/playback links always point to a completed render's output,
even after a re-render fails.

## Goals

- Always show the render parameter form when the user navigates to the render
  page, regardless of whether a previous render exists.
- Display a single-line info banner when a completed render exists:
  "Previously rendered at {date-time stamp}".
- Require confirmation before submitting a re-render when a previous render
  exists. The confirmation dialog includes a parameter summary of the previous
  render.
- Show a submission timestamp on the "submitted" screen so users know when
  their job was submitted.
- Preserve the "submitted" screen behavior for running/queued jobs (no form
  shown while a job is in progress).
- Add `lastCompletedRenderJobId` to the songsets table so download links
  always reference a completed render's output, even if a subsequent re-render
  fails.
- Keep the `RenderComplete` component for future use (landing page from render
  completion notification email), but stop rendering it from the render page.

## Non-Goals

- Do not change the "submitted" screen behavior for running/queued jobs
  (beyond adding the timestamp).
- Do not add client-side render time estimation (use existing server-side
  `estimatedTotalSeconds`).
- Do not change the songset detail page or list page render entry points.
- Do not modify the render-worker pipeline logic (only the DB completion
  function).
- Do not add download buttons to the banner (users download from the songset
  kebab menu).
- Do not use `date-fns` — use native `Intl.DateTimeFormat` for all date-time
  formatting.
- Do not update play pages or `computeRenderState()` to use
  `lastCompletedRenderJobId` (deferred to follow-up PR).

## Current Behavior

### Render page blocks re-render

In `webapp/src/app/songsets/[id]/render/page.tsx:128-133`, when a completed
render job exists and `renderState === "fresh"`, the page auto-redirects to the
`RenderComplete` screen:

```ts
} else if (job.status === "completed") {
  setJobId(job.id)
  setJobData(job)
  if (renderState === "fresh") {
    setScreenState("complete")
  }
}
```

This blocks the user from changing parameters and re-rendering.

### Download links break after failed re-render

When `createRenderJob()` runs, it immediately overwrites
`songsets.latestRenderJobId` to point to the new job. If the new job fails, the
songset detail page's kebab menu download links reference the failed job (which
has no R2 output), making the previous completed render's output inaccessible
through the UI.

## Implementation Steps

### Step 1: Add `lastCompletedRenderJobId` to songsets schema

**File:** `webapp/src/db/schema.ts`

Add a new column to the `songsets` table:

```ts
export const songsets = pgTable("songsets", {
  // ... existing columns ...
  latestRenderJobId: text("latest_render_job_id"),
  lastFailedRenderJobId: text("last_failed_render_job_id"),
  lastCompletedRenderJobId: text("last_completed_render_job_id"),  // NEW
});
```

### Step 2: Generate and write migration

**File:** `webapp/drizzle/0012_*.sql` (new, generated by `npx drizzle-kit generate`)

The migration should include a backfill to populate the new column from existing
data:

```sql
ALTER TABLE songsets ADD COLUMN last_completed_render_job_id text;

UPDATE songsets SET last_completed_render_job_id = latest_render_job_id
  WHERE latest_render_job_id IN (SELECT id FROM render_jobs WHERE status = 'completed');
```

### Step 3: Update render-worker `complete_render_job()`

**File:** `services/render-worker/src/sow_render_worker/db.py`

Add a transactional `UPDATE songsets SET last_completed_render_job_id` inside
`complete_render_job()`, following the same pattern used by `fail_render_job()`:

```python
def complete_render_job(conn, job_id, user_id, mp3_r2_key=None, mp4_r2_key=None, chapters_r2_key=None):
    # ... existing job update logic ...

    conn.autocommit = False
    with conn.cursor() as cur:
        cur.execute(
            "UPDATE render_jobs SET ... WHERE id = %s AND user_id = %s AND status = %s "
            "RETURNING *",
            (...)
        )
        row = cur.fetchone()
    if not row:
        conn.rollback()
        return None

    # NEW: Update songsets.last_completed_render_job_id
    with conn.cursor() as cur:
        cur.execute(
            "UPDATE songsets SET last_completed_render_job_id = %s, updated_at = %s "
            "WHERE id = %s",
            (job_id, now, row["songset_id"]),
        )
    conn.commit()

    return _row_to_render_job(row)
```

Note: The existing `complete_render_job()` does not use transactions (it relies
on autocommit). This change wraps both updates in a transaction, matching the
pattern in `fail_render_job()`.

## Acceptance Criteria

- Navigating to `/songsets/[id]/render` always shows the render parameter form,
  even when a previous render exists with `renderState === "fresh"`.
- When a previous completed render exists, a single-line blue/info banner
  appears above the form: "Previously rendered at {date-time stamp}" (no
  collapse, no parameter summary, no elapsed time, no relative time).
- When no previous render exists, the form appears without the banner (current
  behavior).
- When a render job is running/queued, the "submitted" screen still shows
  (current behavior, unchanged).
- The "submitted" screen shows a submission date-time stamp when `submittedAt`
  is available.
- Clicking "Start Render" when a previous render exists shows a confirmation
  dialog with parameter summary before submitting.
- Clicking "Start Render" when no previous render exists submits directly (no
  confirmation).
- The `RenderComplete` component still exists and compiles but is not rendered
  from the render page.
- `AlertDialog` UI component is installed via shadcn.
- `formatDuration` is extracted to `webapp/src/lib/format.ts`.
- `lastCompletedRenderJobId` column is added to the `songsets` table with
  backfill migration.
- `completeRenderJob()` (both webapp and render-worker) updates
  `songsets.lastCompletedRenderJobId` on job completion.
- Download handlers on songset detail and list pages use
  `lastCompletedRenderJobId` instead of `latestRenderJobId`.
- Kebab menu download buttons are disabled when `lastCompletedRenderJobId` is
  null (instead of `latestRenderJobId`).
- All date-time formatting uses native `Intl.DateTimeFormat` (no `date-fns`
  dependency added).
- All existing tests pass with updated fixtures.
