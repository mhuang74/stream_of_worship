# LLM API Rate-Limit Retry Enhancement — v2

> **Status**: Draft
> **Related**: `specs/llm-rate-limit-retry-v1.md` (introduced 429 retry with exponential backoff), `specs/analysis-service-rate-limit-timeout-tuning-v1.md` (Issue 2 fixed the same anti-pattern for YouTube transcript *fetch* SSL errors)

## Summary

Two related gaps in the LLM retry utility (`workers/llm_rate_limit.py`) cause the YouTube transcript path to be abandoned prematurely and cause avoidable `concurrent_budget_exceeded` 429s:

1. **Transient 5xx errors (notably Cloudflare 524 origin timeout) are not retried** — they propagate on the first attempt, even when the provider's response body explicitly advertises `"retryable": true, "retry_after": 120`. The 8 retry attempts and 300s budget go unused, and the job falls back to expensive ASR.
2. **LLM HTTP calls are not spaced apart** — the `SOW_LLM_MAX_CONCURRENT=3` semaphore matches the provider's 3-slot budget exactly, but it is a *concurrency* limiter, not a *rate* limiter. When one job's HTTP response returns and the slot is released, a waiting job fires within milliseconds, before the provider's `in_flight` counter has decremented. The result is a steady stream of `concurrent_budget_exceeded` 429s that consume retry budget unnecessarily.

## Problem

### Issue 1: 524 Cloudflare origin timeout kills YouTube path on first attempt

#### Root Cause

`_call_llm_with_rate_limit_retry()` in `workers/llm_rate_limit.py:294` only retries when `_is_llm_rate_limited_error(e)` returns `True`. That function (lines 52-104) matches **only** 429 signals:

1. `status_code == 429` (or `code == 429`)
2. Exception type name containing `RateLimitError` / `rate_limit`
3. Strings containing `429`, `rate_limit`, `rate_limit_error`, `concurrent_budget_exceeded`

A 524 matches **none** of these. So at `llm_rate_limit.py:328-331`:

```python
except Exception as e:
    if not _is_llm_rate_limited_error(e):
        raise  # ← fires here, zero retries
```

The exception propagates immediately. Back in `_llm_correct()` at `youtube_transcript.py:809-814`, the same check returns `False`, so it raises `YouTubeTranscriptError("LLM correction failed: ...")`. Back in `_attempt_youtube_transcript_path()` at `lrc.py:912-918`, the `except Exception as e:` block logs "YouTube transcript path FAILED" and returns `None`, triggering the ASR fallback.

#### Why this is unnecessary

- The YouTube transcript was already fetched successfully — only the LLM correction (one HTTP call to align timestamps) failed
- There were **8 retry attempts budgeted** (`SOW_LLM_RATE_LIMIT_MAX_RETRIES=8`) and **300s of timeout budget** (`SOW_LLM_RATE_LIMIT_TIMEOUT_SECONDS=300`) — ample room for one 120s `retry_after`
- The provider itself marked the error `retryable: true` — the system just doesn't read that field for 5xx
- A single retry after ~120s (within budget) would almost certainly have succeeded

#### Prior awareness

The spec `specs/analysis-service-rate-limit-timeout-tuning-v1.md` (Issue 2) documented the **same anti-pattern** in the YouTube *fetch* path (SSL errors not retried, only 429 was) and fixed it by adding `_is_transient_connection_error()`. But the **LLM correction path** has the identical blind spot and was not addressed. Issue 3 in that same spec even notes "LLM rate limiting is healthy; no tuning needed" — based on a batch with zero 524s.

### Issue 2: LLM calls spaced too closely → avoidable 429s

Queue state at the same moment: `LRC[processing:7] STEM_SEPARATION[processing:2]` — 7 LRC jobs competing for 3 LLM slots.

#### Root Cause

`SOW_LLM_MAX_CONCURRENT = 3` (in `config.py:97`) deliberately matches the provider's 3-slot concurrency budget. But it is a **concurrency limiter, not a rate limiter**:

- When one job's HTTP response returns and its semaphore slot is released, a waiting job fires its HTTP request within milliseconds
- The provider's `in_flight` counter may lag for a few hundred milliseconds after the HTTP response is received (eventual consistency on their side)
- The new request arrives while the provider still counts the previous request as in-flight → `concurrent_budget_exceeded` 429

The retry loop then compounds the problem: each retry from THIS job fires a new HTTP request while 2 OTHER jobs' requests may still be in flight at the provider. The provider's counter: 2 ongoing + this retry = 3, which is fine in steady state, but timing skew causes transient over-counts.

The YouTube transcript path already solves this exact problem with `_enforce_min_interval()` at `youtube_transcript.py:224-248` — an `asyncio.Lock`-guarded module-level timestamp that forces a configurable minimum delay between requests. The LLM path has no equivalent throttle.

## Design

### Principle

1. **Retry transient 5xx errors** with the same exponential backoff used for 429s, honoring the provider's `retry_after` field. Reuse the existing 8-attempt / 300s budget — no separate counter.
2. **Pace LLM HTTP calls** with a module-level min-interval throttle, mirroring the proven `_enforce_min_interval()` pattern from the YouTube transcript rate limiter. The throttle fires *while holding the LLM semaphore slot*, so it paces active requests without blocking idle jobs.

### Part A — Make transient 5xx errors retryable

**File: `ops/analysis-service/src/sow_analysis/workers/llm_rate_limit.py`**

#### A.1 Add `_is_llm_retryable_error(e: Exception) -> bool`

Sibling to `_is_llm_rate_limited_error`. Uses the same exception-cause-chain walk pattern (visited set + `__cause__`/`__context__` traversal). Detection strategies:

1. **Status code check**: `getattr(exc, "status_code", None)` in `{500, 502, 503, 504, 520, 521, 522, 523, 524, 529}` (also check `getattr(exc, "code", None)` for symmetry with the 429 path)
2. **Type-name check**: `type(exc).__name__` in `{"APITimeoutError", "APIConnectionError", "APIStatusError"}` (OpenAI SDK exception types)
3. **Provider JSON body check**: parse the exception message/body via the existing `_extract_json_from_text` helper, then check for top-level `"retryable": true`. The 524 response body has `retryable` at the top level (not nested in `error.{...}`), and `_extract_json_from_text` already handles the OpenAI SDK's `"Error code: 524 - {...}"` format.

Returns `False` for parse errors (`ValueError`, `json.JSONDecodeError`) and other non-transient errors — these must continue to propagate immediately.

#### A.2 Update the retry loop in `_call_llm_with_rate_limit_retry`

At `llm_rate_limit.py:328-331`, change the propagation logic:

```python
except Exception as e:
    is_rate_limit = _is_llm_rate_limited_error(e)
    is_retryable = _is_llm_retryable_error(e)
    if not (is_rate_limit or is_retryable):
        # Non-retriable error — propagate immediately
        raise

    last_exception = e

    # Don't sleep after the last attempt
    if attempt >= max_attempts - 1:
        break

    # ... existing budget check, retry_after extraction, backoff computation ...
```

The existing `_extract_retry_after` already parses top-level `retry_after` correctly (via `data.get("error", data)` fallback at `llm_rate_limit.py:175`), so the 524's `retry_after: 120` will be honored as a minimum delay. No changes needed to the backoff machinery.

#### A.3 Differentiate log messages

Update the backoff warning log so operators can distinguish 429 retries from 5xx retries:

- Existing (429 path): `"LLM rate limited (429) for %s, attempt %d/%d, backing off %.1fs ..."`
- New (5xx path, when `is_retryable and not is_rate_limit`): `"LLM transient error (5xx) for %s, attempt %d/%d, backing off %.1fs ..."`

Both share the same backoff parameters (`retry_after`, budget remaining, etc.).

#### A.4 No changes needed in `youtube_transcript.py` or `lrc.py`

`_llm_correct()` at `youtube_transcript.py:809-814` raises `YouTubeTranscriptError` only if `_call_llm_with_rate_limit_retry` propagates an exception. With the new retry logic, retryable 5xx errors are retried inside the wrapper and never reach this `except` block. The propagation chain is intact — only the retry decision changes.

`_attempt_youtube_transcript_path()` at `lrc.py:912-918` only fires the ASR fallback if `youtube_transcript_to_lrc` actually raises, which the new retry logic prevents for transient 5xx.

### Part B — Add min-interval throttle between LLM calls

**File: `ops/analysis-service/src/sow_analysis/workers/llm_rate_limit.py`**

#### B.1 Add module-level throttle state

Alongside `_llm_semaphore` (line 26):

```python
_llm_interval_lock: Optional[asyncio.Lock] = None
_llm_last_request_time: float = 0.0
```

Lazy-initialize the `asyncio.Lock` on first use (same pattern as `_llm_semaphore` at lines 36-43).

#### B.2 Add `_enforce_llm_min_interval()`

Mirror `_enforce_min_interval()` in `youtube_transcript.py:224-248`:

```python
async def _enforce_llm_min_interval() -> None:
    """Sleep if the last LLM HTTP request was too recent (min-interval throttle).

    Uses an asyncio.Lock to serialize the timestamp check so that
    concurrent callers are spaced out correctly.
    """
    global _llm_interval_lock, _llm_last_request_time

    if _llm_interval_lock is None:
        _llm_interval_lock = asyncio.Lock()

    min_interval = settings.SOW_LLM_MIN_INTERVAL_SECONDS
    if min_interval <= 0:
        return

    async with _llm_interval_lock:
        now = time.monotonic()
        elapsed = now - _llm_last_request_time
        if elapsed < min_interval:
            # Add jitter (±25%) to prevent thundering herd when multiple
            # jobs wake from cooldown simultaneously
            jitter = min_interval * 0.25
            wait = (min_interval - elapsed) + random.uniform(-jitter, jitter)
            if wait > 0:
                logger.debug(
                    "LLM min-interval throttle: spacing request, sleeping %.2fs", wait
                )
                await asyncio.sleep(wait)
        _llm_last_request_time = time.monotonic()
```

#### B.3 Call the throttle inside the retry loop

In `_call_llm_with_rate_limit_retry`, at the top of the `for attempt in range(max_attempts):` loop body, **before** `loop.run_in_executor(None, sync_fn)`:

```python
for attempt in range(max_attempts):
    await _enforce_llm_min_interval()  # ← NEW: pace requests across jobs
    try:
        return await loop.run_in_executor(None, sync_fn)
    except Exception as e:
        # ... existing retry logic ...
```

**Placement rationale**: This happens *while* the caller is already holding the LLM semaphore slot (callers wrap with `_acquire_llm_slot()` / `_release_llm_slot()`), so we don't throttle idle jobs waiting on the semaphore. The throttle paces both initial attempts AND each retry, but NOT during backoff sleep (which the loop already handles separately via `await asyncio.sleep(delay)`).

#### B.4 Add config setting

**File: `ops/analysis-service/src/sow_analysis/config.py`** — Add near `SOW_LLM_MAX_CONCURRENT` (line 97):

```python
SOW_LLM_MIN_INTERVAL_SECONDS: float = 2.0
# Minimum gap (seconds) between consecutive LLM HTTP calls across all jobs.
# Compensates for provider-side in_flight accounting lag — when one call's
# HTTP response returns and the semaphore slot is released, the next job
# fires immediately, but the provider may still count the previous request
# as in-flight for a few hundred ms. This throttle prevents the resulting
# concurrent_budget_exceeded 429s. Set to 0 to disable.
```

**Default value rationale**: 2.0s gives ~1-2s headroom after a request completes for the provider's `in_flight` accounting to settle. Allows ~0.5 req/s sustained per slot (1.5 req/s across 3 slots), well below provider capacity. Conservative choice — near-eliminates `concurrent_budget_exceeded` at the cost of slightly higher latency when many jobs queue up.

**No free-mode variant**: The LLM provider is the same in free and paid mode (unlike the YouTube transcript API which has separate free-tier quotas). A single setting suffices.

#### B.5 Exclude the health check

The periodic LLM health check at `routes/health.py:68` calls `client.chat.completions.create(...)` directly, bypassing both the semaphore and the rate-limit wrapper. It runs infrequently and should respond quickly. **Do not wrap it** with `_enforce_llm_min_interval` — doing so would add delay to status reporting without benefit.

## What Does NOT Change

- `SOW_LLM_MAX_CONCURRENT = 3` (semaphore still matches provider budget)
- `SOW_LLM_RATE_LIMIT_MAX_RETRIES = 8` (same budget for 429 and 5xx)
- `SOW_LLM_RATE_LIMIT_TIMEOUT_SECONDS = 300` (same wall-clock budget)
- `SOW_LLM_RATE_LIMIT_BASE_DELAY = 2.0` / `SOW_LLM_RATE_LIMIT_MAX_DELAY = 30.0` (same backoff curve)
- `_extract_retry_after` / `_extract_backoff_config` (already handle top-level `retry_after` and `retry_strategy`)
- Circuit breaker logic (LLM path has none; YouTube transcript path's circuit breaker is unaffected)
- The three-tier LRC fallback cascade (Tier 1 YouTube → Tier 2 Qwen3 ASR → Tier 3 Whisper) — only the retry behavior within Tier 1's LLM correction step changes
- Health check at `routes/health.py:68` (excluded from throttle by design)
